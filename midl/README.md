# Motherboard IDL

midl (Motherboard Interface Definition Language) is a language for defining service interfaces and generating clients and server runtime boilerplate.

## Example
```
// comments use double slashes
// triple slashes are documentation in rustdoc format.


// service keyword for declaring service interfaces
service Init {
	// store keyword declare a store with a type
	// types:
	// - u8..u64
	// - i8..i64
	// - string
	// - bool
	// - char
	// - f32
	// - f64
	// - T[] for arrays (dynamic)
	// - (..., ...) for tuples
	// - AnonymousStore<T> are stores that can be returned from functions and just like stores can be subscribed to. The main difference from normal stores is that they don't have a name and allows the service to have stores with dynamic parameters such as IDs for certain objects, having more fine grained control over who accesses what. After all store subscriptions are cleaned up, the anonymous store is also cleaned up and can't subscribed again.
	// stores can have a `public` modifier, but in this case, they are private because i want to check if it's TIBS using those stores
	store totalComponents: string[],
	store startingComponents: string[];
	store startedComponents: string[];
	store logs: string[];
}


service Auth {
	type AuthSessionToken = string;
	type PID = u64;

	enum AuthSessionStatus {
		Pending,
		LoggedIn,
		Canceled
	}


	enum FarProcessState {
		Running,
		ExittedNormally,
		Crashed { logs: string }
	}

	/// This starts a new login session and should be called as soon as the login screen shows the login prompt.
	/// Login sessions are scoped per process, so a successful login on one process does not allow another process to run commands as that user.
	///
	/// This is needed to tell the auth service when the prompt appears so it can trigger side effects like turning on the fingerprint sensor
	/// for passwordless login.
	///
	/// ## Concurrent sessions
	///
	/// Depending on the situation, only one auth session can be active at a time because otherwise they would share a global resource or authentication source
	/// like a fingerprint sensor, in those cases, the `Conflict` error may be returned. So if you decided you want to login into a different user, don't forget to first
	/// cancel the old session id before starting a new one to avoid issues.
	fn start_user_auth(user: string) -> AuthSessionToken?(error { UserNotFound, Conflict });

	/// Resolves a login session with a password, returns true if the password matches, false otherwise. This requires `Auth.start_user_auth(user)` to be called first
	fn login_with_password(sessionId: AuthSessionToken, password: string) -> bool?(error { InvalidSessionId });

	/// Runs a command as a before authorized user
	/// The process generated by this command will be ran in the background and these are called "far processes" since they run as another user.
	/// Far processes can be watched with `Auth.watch_far_process(sessionId, pid)` and only the process who created the session can watch the far process it spawned.
	fn run_command_as(sessionId: AuthSessionToken) -> PID?(error { InvalidSessionId, CommandSpawnFailed });

	/// Closes the login session, not resulting in a successful login.
	fn cancel_login(sessionId: AuthSessionToken) -> void?(error { NotFound });

	/// Returns a anonymous store that contains the current state of the auth session which can be `Pending`, `LoggedIn` or `Canceled`.
	fn auth_session_status(sessionId: AuthSessionToken) -> AnonymousStore<AuthSessionStatus>?(error { InvalidSessionId, PermissionDenied });

	/// Watches the status of a far process.
	///
	/// See [`Auth.run_command_as`]
	fn watch_far_process(sessionId: AuthSessionToken, pid: PID) -> AnonymousStore<FarProcessState>?(error { InvalidSessionId, PermissionDenied, PidNotFound });
}
```




## Setup code generation

Add this to your build.rs:


### Client

```rust
use std::{env, fs, path::PathBuf};

fn main() {
    let manifest_dir = PathBuf::from(env::var_os("CARGO_MANIFEST_DIR").unwrap());
    let idl_path = manifest_dir.join("interface.midl");
    println!("cargo:rerun-if-changed={}", idl_path.display());
    let source = fs::read_to_string(&idl_path).unwrap();
    let document = midl::parse_document(&source).unwrap();
    let client_bindings = midl::generate_rust_with_mode(&document, midl::RustMode::Client);
    let out_dir = PathBuf::from(env::var_os("OUT_DIR").unwrap());
    fs::write(out_dir.join("client_bindings.rs"), client_bindings).unwrap();
}
```

Then you can include the generated code as a rust module:

```
#[allow(dead_code, non_snake_case, unused_imports, unused_variables)]
pub mod client_bindings {
    include!(concat!(env!("OUT_DIR"), "/client_bindings.rs"));
}
```
